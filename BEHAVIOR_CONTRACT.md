# Application Behavior Contract

This file defines the exact contract for what the application behavior system
generates.

It is the narrowing document for
[BEHAVIOR_SYSTEM.md](/Users/arpit/Developer/flow-state/BEHAVIOR_SYSTEM.md):
the backlog says what to build, while this file says what the outputs must look
like and what must not be generated.

## Core rule

Generate one canonical machine-owned contract artifact.

Everything else should be:

- a renderer over that contract
- a diff between two contracts
- or an explicitly requested user-owned scaffold

If a fact can be rendered from the canonical contract, do not commit a second
generated artifact that repeats the same fact in another format.

## Canonical generated artifact

Committed, machine-owned, deterministic:

```txt
apps/docs/src/generated/behavior-contract.json
```

## What is not a separate committed generated artifact

Do not create separate committed generated files for:

- summary Markdown
- coverage Markdown
- diff reports
- PR review notes

Those should render from the canonical contract at docs-build time, CLI time,
or review time.

## Minimal command surface

The minimal product should expose one namespace with as few commands as
possible:

```txt
flow-state behavior build
flow-state behavior render
flow-state behavior diff
```

Rules:

- `build` writes the canonical JSON artifact.
- `render` prints one shared brief by default.
- `render --module <id>` renders a filtered module slice from the canonical
  app contract.
- `render --section coverage` prints the detailed coverage view.
- `diff` compares two contracts or two build targets.
- `diff --module <id>` compares the same module slice across two app contracts.
- Do not add separate `explain`, `summary`, or `ai-context` commands.
- Optional `scaffold` work is future and should only be added if the three core
  commands prove insufficient.

## Build input surface

The build root is the assembled app, not a bag of peer module inputs.

```ts
type FlowBehaviorBuildTarget = Readonly<{
  app: FlowAppDefinition<ReadonlyArray<FlowModuleDefinition>>;
  stories?: ReadonlyArray<FlowStoriesDescriptor>;
}>;
```

Rules:

- `app` is the only required root input.
- `app` must come from a real `flow.app({ modules: [...] })` assembly.
- The builder must derive modules, resources, transactions, machines, streams,
  and views from `app.modules`.
- Do not require callers to pass parallel `modules`, `resources`,
  `transactions`, `machines`, `streams`, or `views` lists once those facts are
  already reachable from `app`.
- Stories may stay explicit in v1 until Flow State has a first-class app-level
  story registry.
- The recommended client-side entrypoint is one explicit gateway file such as
  `src/app/behavior.ts`.

## Gateway assumptions

The client-side integration surface should stay explicit and small.

```ts
type FlowBehaviorGateway = Readonly<{
  app: FlowAppDefinition<ReadonlyArray<FlowModuleDefinition>>;
  stories?: ReadonlyArray<FlowStoriesDescriptor>;
}>;
```

Rules:

- Prefer one explicit named export such as `BehaviorGateway` from
  `src/app/behavior.ts`.
- If the CLI owns gateway loading, prefer an explicit entry-path flag and use
  `src/app/behavior.ts` as the docs/examples default instead of repo-wide
  autodiscovery.
- v1 should load one explicit gateway module path, not scan the repo looking
  for apps, modules, or stories.
- Do not add filesystem crawling, export-name guessing, AST scraping, or
  component-tree inference to discover behavior inputs.
- `stories` stay explicit until `flow.app(...)` owns an app-level story
  registry.

## Module slice mode

Module-level work is supported as a focused lens over the app contract, not as
a competing canonical artifact.

Rules:

- App is truth.
- Module is lens.
- Module slice mode must always derive from the assembled app context.
- Module slice mode must not require a second module-only committed contract
  file.
- Module slices may be rendered or diffed, but the canonical committed artifact
  stays app-level.
- If a module depends on shared app facts, the slice should preserve those
  relationships honestly instead of pretending the module is fully standalone.

## User-owned scaffolds

These are not generated by default.

They are only created when a user explicitly asks for scaffolds:

```txt
src/__flow__/*.story.ts
src/__flow__/*.contract.test.ts
src/__flow__/*.scenario.ts
```

These files are generated once, then owned by the user. They must never be
treated as canonical sources of truth over the behavior contract itself.

## Canonical JSON schema

The canonical contract must be JSON-safe and deterministic. It must not depend
on timestamps, process ids, hash salts, or non-stable object-key ordering.

```ts
type FlowBehaviorContract = Readonly<{
  version: "flow-state/behavior-contract.v1";
  app: FlowBehaviorApp;
  modules: ReadonlyArray<FlowBehaviorModule>;
  resources: ReadonlyArray<FlowBehaviorResource>;
  transactions: ReadonlyArray<FlowBehaviorTransaction>;
  machines: ReadonlyArray<FlowBehaviorMachine>;
  streams: ReadonlyArray<FlowBehaviorStream>;
  views: ReadonlyArray<FlowBehaviorView>;
  stories: ReadonlyArray<FlowBehaviorStory>;
}>;
```

## Contract sections

### `app`

```ts
type FlowBehaviorApp = Readonly<{
  id: string;
  moduleIds: ReadonlyArray<string>;
}>;
```

Requirements:

- `id` comes from the real app descriptor.
- `moduleIds` preserve app assembly order.
- This is the root projection of the assembled app boundary.
- Do not imply broader app-level validation than the currently proved module
  assembly, inventory, ownership, and selective duplicate checks.

### `modules`

```ts
type FlowBehaviorModule = Readonly<{
  id: string;
  dependencies: ReadonlyArray<string>;
  screenIds: ReadonlyArray<string>;
  tagIds: ReadonlyArray<string>;
  fixtureIds: ReadonlyArray<string>;
}>;
```

Requirements:

- This is a normalized projection of `flow.module(...)` and `App.inventory()`.
- It must not guess hidden domain groupings from filenames.
- These are derived child records under the app root, not a second build root.
- `screenIds` are coarse module metadata and must not be treated as a precise
  screen-to-view or component ownership map.
- `dependencies` and `tagIds` are declared metadata; expose them honestly
  without pretending they already carry stronger runtime semantics than they do
  today.

### `resources`

```ts
type FlowBehaviorResource = Readonly<{
  id: string;
  moduleId: string | null;
  hasSchema: boolean;
  hasPlaceholder: boolean;
  freshness: null | Readonly<{
    staleAfter: string | number;
    onInvalidate: "active" | "lazy" | "never" | null;
  }>;
}>;
```

Requirements:

- Represent only facts we can read from live resource descriptors today.
- Do not pretend dynamic tag outputs or lookup behavior were statically
  enumerated if they were not.

### `transactions`

```ts
type FlowBehaviorTransaction = Readonly<{
  id: string;
  moduleId: string | null;
  hasParams: boolean;
  hasPreview: boolean;
  hasInvalidates: boolean;
  hasQueueWhen: boolean;
  hasQueueReplay: boolean;
  hasQueueUndo: boolean;
  concurrency: string | null;
  routeKinds: ReadonlyArray<"success" | "failure" | "defect" | "interrupt">;
}>;
```

Requirements:

- Represent the real descriptor surface only.
- Do not imply queue persistence, replay guarantees, or undo semantics beyond
  what the current status page already supports.

### `machines`

```ts
type FlowBehaviorMachine = Readonly<{
  id: string;
  moduleId: string | null;
  initialStateId: string;
  states: ReadonlyArray<FlowBehaviorState>;
  transitions: ReadonlyArray<FlowBehaviorTransition>;
}>;

type FlowBehaviorState = Readonly<{
  id: string;
  terminal: boolean;
  childIds: ReadonlyArray<string>;
  timedTransitions: ReadonlyArray<
    Readonly<{
      id: string;
      delay: unknown;
      target: string;
    }>
  >;
  eventlessTransitions: ReadonlyArray<
    Readonly<{
      id: string;
      target: string;
    }>
  >;
}>;

type FlowBehaviorTransition = Readonly<{
  id: string;
  source: string;
  target: string;
  eventType: string;
}>;
```

Requirements:

- Reuse `graphOf(machine)` facts instead of re-deriving machine shape from raw
  config by hand.
- Preserve graph ids so coverage and diff outputs can reference them stably.

### `streams`

```ts
type FlowBehaviorStream = Readonly<{
  id: string;
  moduleId: string | null;
  hasParams: boolean;
  pressure: null | Readonly<{
    strategy: "queue" | "coalesce-latest";
    limit: number | null;
  }>;
  routeKinds: ReadonlyArray<"value" | "done" | "failure" | "defect" | "interrupt">;
}>;
```

Requirements:

- Only include pressure details that are statically visible today.
- Do not fabricate diagnostics counters or runtime stats that belong to future
  tooling.

### `views`

```ts
type FlowBehaviorView = Readonly<{
  id: string;
  moduleId: string | null;
  sources: ReadonlyArray<
    | "context"
    | "resources"
    | "transactions"
    | "streams"
    | "timers"
    | "children"
    | "issues"
    | "receipts"
  >;
}>;
```

Requirements:

- v1 is source-oriented, not field-oriented.
- If we later want field-level or case-level view docs, that needs additive
  metadata and must not be faked from function source parsing.

### `stories`

```ts
type FlowBehaviorStory = Readonly<{
  id: string;
  machineId: string;
  title: string;
  tags: ReadonlyArray<string>;
  start: "default" | "snapshot" | "setup";
  expectedState: string | null;
  seed: null | Readonly<{
    resourceCount: number;
    fixtureIds: ReadonlyArray<string>;
    hasBoot: boolean;
    actorId: string | null;
  }>;
  expectedFacts: Readonly<{
    receiptTypes: ReadonlyArray<string>;
    relatedIds: ReadonlyArray<string>;
    issueKinds: ReadonlyArray<string>;
    issueSources: ReadonlyArray<string>;
    outcomeKinds: ReadonlyArray<string>;
    outcomeSources: ReadonlyArray<string>;
  }>;
}>;
```

Requirements:

- Reuse `flowStories(...)` and story-doc facts where possible.
- Preserve whether a story is blocked by setup-description-only start.
- Keep stories as explicit curated proof inputs; do not invent story discovery
  by scanning files or parsing component code.

## Rendering templates

These are render contracts over `behavior-contract.json`. They are not separate
committed generated sources.

### Shared brief

Command:

```txt
flow-state behavior render
flow-state behavior render --module <module id>
```

Output template:

```md
# <App label>

## App

- Modules: <comma-separated module ids>
- Screens: <comma-separated screen ids>
- Fixtures: <comma-separated fixture ids or "none">

## Main Machines

### <machine id>

- Initial state: <state id>
- States: <comma-separated state ids>
- Key transitions:
  - <source> --<eventType>--> <target>

## Runtime Work

### <machine id>/<state id>

- Children: <child ids or "none">
- Timed transitions: <timer ids or "none">
- Eventless transitions: <target ids or "none">

## Writes And Streams

- Transactions: <transaction ids with preview/queue/concurrency facts>
- Streams: <stream ids with pressure/route facts>

## Views

- <view id>: <sources>

## Current Proof Surface

- Stories: <count>
- Covered states/transitions: <summary computed at render time>
- Blocked or mismatch stories: <summary>
```

Rules:

- This brief is intentionally shared by humans and AI agents.
- Do not generate a second audience-specific summary unless a real deficiency is
  proven later.
- If a caller needs a shorter snippet, derive it from this same brief.
- When `--module <id>` is provided, render the same brief shape filtered to that
  module's reachable records and label it clearly as a module slice.

### Docs brief page

Page sections must render in this order:

1. App
2. Modules
3. Screens
4. Machines
5. Runtime work by state
6. Transactions and streams
7. Views
8. Stories and coverage
9. Links to deeper owner docs

### Detailed coverage view

Command:

```txt
flow-state behavior render --section coverage
flow-state behavior render --module <module id> --section coverage
```

Coverage is derived at render time from:

- machines
- graph ids
- stories
- story coverage logic

It is not stored as a second canonical section in `behavior-contract.json`.

When `--module <id>` is provided, the coverage view must filter to that module's
machines, stories, and derived obligations while preserving honest labels for
shared app context.

Output sections must render in this order:

1. Coverage scope note
2. Covered states by machine
3. Uncovered states by machine
4. Covered transitions by machine
5. Uncovered transitions by machine
6. Covered issue lanes
7. Covered outcome lanes
8. Blocked stories
9. Mismatch stories

### Diff report

Diff output between two contracts must render in this order:

1. App summary
2. Module changes
3. Machine/state/transition changes
4. Resource changes
5. Transaction changes
6. Stream changes
7. View changes
8. Story changes
9. Coverage obligation changes computed from each contract's derived coverage

When `--module <id>` is provided, diff the filtered module slice from each app
contract instead of inventing a separate module-level baseline artifact.

## Determinism rules

- Sort arrays by stable ids unless source order is semantically meaningful.
- Preserve app module order where assembly order matters.
- Do not emit timestamps in committed artifacts.
- Do not emit absolute file paths in the canonical contract.
- Do not include source-code strings, function bodies, or serialized closures.

## Honesty rules

- Do not mark future or partial behavior as generated fact.
- Do not emit queue/replay/undo guarantees beyond statically visible descriptor
  facts plus already-supported status wording.
- Do not emit field-level view docs unless the view definition exposes field
  metadata explicitly.
- Do not emit guard descriptions unless guard metadata exists explicitly.
- Do not create separate AI-only renders when the shared brief already says the
  same thing.

## Review checklist

Every implementation of this contract should be reviewable against this list:

- [ ] Exactly one canonical committed generated contract artifact exists.
- [ ] The canonical artifact is JSON-safe and deterministic.
- [ ] Docs brief renders from the canonical contract instead of duplicating
      data in a second generated source.
- [ ] Coverage is derived at render time instead of becoming a second canonical
      format.
- [ ] Humans and AI share one brief instead of separate generated summaries.
- [ ] Diffing compares contracts, not ad hoc text files.
- [ ] Optional user-owned scaffolds are opt-in and clearly non-canonical.
