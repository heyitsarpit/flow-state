# Proposed Public CLI Shape

This file is the concrete proposed public CLI shape for Flow State's debugging,
inspection, and behavior tooling.

It is meant to answer four questions up front:

1. What are the top-level command families?
2. What should the subcommands and important flags be called?
3. Which existing functions should each command reuse?
4. Where do we still need thin composition code versus genuinely new lower-level code?

## Top-Level Rule

Keep the durable public surface job-first:

- `flow-state behavior ...`
- `flow-state story ...`
- `flow-state trace ...`

## Packaging And Distribution

- Users install the single `flow-state` package.
- Library imports remain:
  `flow-state`,
  `flow-state/react`,
  `flow-state/testing`,
  `flow-state/server`,
  and `flow-state/inspect`.
- The installed `flow-state` package should expose the `flow-state` binary via
  `packages/flow-state/package.json#bin`.
- The CLI is not a separate package and not a public import route such as
  `flow-state/cli`.
- Durable CLI implementation should live under
  `packages/flow-state/src/cli/**` with entrypoint `src/cli/index.ts`.
- Built CLI output should land under `packages/flow-state/dist/cli/**` with the
  installed binary entrypoint at `dist/cli/index.mjs`.
- Durable CLI tests should live under
  `packages/flow-state/src/cli-test/**`.
- `packages/flow-state/scripts/**` remains for repo-local build, proof, and
  maintenance helpers that are not part of the installed CLI contract.

Do not make `inspect` the public top-level family.

Inside those families, preserve meaningful existing transformation names when
they already describe a distinct user-facing job.

When an existing helper name misstates the actual job, rename both the public
CLI spelling and the helper toward the clearer job name instead of freezing a
misleading label into the durable surface.

That means:

- keep job-first families
- let CLI verbs mirror real user-facing jobs like
  `describeStory(...)` (current `storyToDoc(...)`) and
  `summarizeTrace(...)`
- when a helper only adds a mode over the same job, prefer a flag like
  `story run --check` or `trace summarize --contextualize`
- do not mirror every helper one-to-one

## Recommended Tree

```text
flow-state
  behavior
    build
      --project-root <path>
      --gateway <path>
      --output <path>
    render
      --input <path>
      --project-root <path>
      --gateway <path>
      --section <contract|coverage>
      --module <moduleId>
      --format <text|json>
    diff
      --left-input <path>
      --right-input <path>
      --left-project-root <path>
      --left-gateway <path>
      --right-project-root <path>
      --right-gateway <path>
      --module <moduleId>
      --format <text|json>

  story
    list
      --project-root <path>
      --gateway <path>
      --machine <machineId>
      --tag <tag>
      --format <text|json>
    run <story-id>
      --project-root <path>
      --gateway <path>
      --format <pretty|compact|json>
      --check
      --pending-work
      --transcript
      --transactions
      --save-proof <path>
      --save-trace <path>
    describe <story-id>
      --project-root <path>
      --gateway <path>
      --format <text|json>
    paths
      --project-root <path>
      --gateway <path>
      --machine <machineId>
      --strategy <shortest|simple>
      --event <json>
      --from-state <state>
      --to-state <state>
      --limit <n>
      --format <text|json>
      --check

  trace
    summarize <trace-or-proof>
      --format <pretty|compact|json>
      --contextualize
        --project-root <path>
        --gateway <path>
        --machine <machineId>
    diff <left> <right>
      --format <text|json>
      --section <name>
    proof <trace-or-proof>
      --actor <actorId>
      --correlation <correlationId>
      --issues
      --timeline
      --format <pretty|json>
```

## Why These Names

### `behavior`

- `build`, `render`, and `diff` already match the live script and the existing
  behavior helper surface.
- Coverage should stay under `render --section coverage` instead of becoming a
  separate top-level command.

### `story`

- `story run` preserves `runFlowStory(...)`.
- `story run --check` is narrower than a separate `story check` command because
  expectation checking is still the same execution workflow.
- `story describe` matches what the helper actually returns: a normalized story
  description, not a rendered document.
- `story list` stays the discovery/index command over the declared stories.
- `story paths` gives `test.model(machine)` a durable CLI home for
  guard-aware path discovery and event-sequence validation.
- Model paths belong here as a discovery companion to live scenarios.
  They do not replace `story run` or runtime harness evidence.
- `story paths --strategy shortest|simple` is narrower than separate sibling
  commands because the real distinction is search strategy, not job identity.
- `story paths --check` keeps exact-sequence validation in the same path
  workflow instead of splitting it into a second mini-surface.
- Do not add a separate `story trace` command. That would duplicate `story run`
  output and the `trace` family.

### `trace`

- `trace summarize` mirrors `summarizeTrace(...)` more directly than `summary`.
- `trace diff` mirrors `diffTrace(...)`.
- `trace summarize --contextualize` is narrower than a separate
  `trace contextualize` command because context attachment is enrichment over
  the same trace overview workflow.
- `trace proof` is clearer than `view` because the current reusable bundle is a
  local inspection proof with actor/correlation/timeline slices.

## Reuse Map

### `behavior build`

Primary reuse:

- `buildBehaviorContract(...)`

Supporting reuse:

- `.inventory()`
- `graphOf(...)`
- `describeStory(...)` (current `storyToDoc(...)`)

### `behavior render`

Primary reuse:

- `renderBehaviorContract(...)`
- `renderBehaviorCoverage(...)`

Supporting reuse:

- `sliceBehaviorContract(...)`

### `behavior diff`

Primary reuse:

- `diffBehaviorContracts(...)`
- `renderBehaviorDiff(...)`

### `story list`

Primary reuse:

- `flowStories(...)`
- `describeStory(...)` (current `storyToDoc(...)`)

Notes:

- `describeStory(...)` is the best existing normalized story shape for listing,
  filters, and docs-oriented text output.

### `story run`

Primary reuse:

- `runFlowStory(...)`
- `checkStory(...)` (current `storyToTest(...)`) when `--check` is present

Supporting reuse:

- `formatTrace(...)`
- `formatTracePretty(...)`
- `receiptSummary()`
- `issueSummary()`
- `pendingWork()`
- `formatPendingWorkPretty(...)`
- `formatScenarioTranscript(...)`
- `formatTransactionEventsPretty(...)`
- `createTraceProof(...)` (current `createLocalInspectionProof(...)`)
- `exportTraceArtifact(...)`

Notes:

- Default output should be the run result.
- `--check` should add the expectation-check delta over the same run result
  instead of acting like a second viewer.
- Default text and JSON output should include compact receipt and issue summaries
  so agents get assertable runtime facts without opting into a larger transcript.
- `--pending-work` should expose the stuck-run diagnostic surface in both
  machine-readable and pretty forms.
- `--transcript` should reuse `formatScenarioTranscript(...)`.
- `--transactions` should reuse `formatTransactionEventsPretty(...)`.
- `--save-proof` should reuse `createTraceProof(...)`.
- `--save-trace` should reuse `exportTraceArtifact(...)`.

### `story describe`

Primary reuse:

- `describeStory(...)` (current `storyToDoc(...)`)

Supporting reuse:

- `flowStories(...)`

### `story paths`

Primary reuse:

- `test.model(machine).getShortestPaths(...)` when `--strategy shortest`
- `test.model(machine).getSimplePaths(...)` when `--strategy simple`
- `graphOf(machine).pathFromEvents(...)` when `--check` is present

Supporting reuse:

- gateway machine lookup
- `test.model(machine)` as the owning mental model for path exploration

Notes:

- This is the CLI home for guard-aware path discovery and validation.
- `--strategy shortest|simple` is for discovery.
- `--check` answers "is this exact event sequence legal, and if so what path
  does it produce?"
- It should not start a live runtime scenario on its own or print runtime
  receipts/trace evidence.

### `trace summarize`

Primary reuse:

- `summarizeTrace(...)`
- `contextualizeTrace(...)` (current `analyzeTrace(...)`) when
  `--contextualize` is present

Supporting reuse:

- `formatTrace(...)`
- `formatTracePretty(...)`
- `formatResourceFreshnessReport(...)`
- `formatTransactionOverlapSummary(...)`
- `formatRehydrationSummary(...)`

Notes:

- Default output should stay a whole-trace overview.
- `--contextualize` should add machine-aware annotation over that same overview
  rather than acting like a second standalone summary surface.

### `trace diff`

Primary reuse:

- `diffTrace(...)`

Notes:

- `--section` names should mirror the real diff sections already produced by
  the structured diff descriptor.

### `trace proof`

Primary reuse:

- `createTraceProof(...)` (current `createLocalInspectionProof(...)`)
- `formatTrace(...)`
- `formatTracePretty(...)`
- `formatInspectionTimeline(...)`
- `formatInspectionTimelinePretty(...)`

Supporting reuse:

- proof actor tree
- proof correlation detail
- proof formatted snapshots

Selector mapping:

- `--actor <actorId>`: actor-focused slice
- `--correlation <correlationId>`: correlation-focused slice
- `--issues`: issue-focused slice
- `--timeline`: live event timeline view

## Library-First Surfaces That Should Not Become Top-Level Commands

These are important and useful, but they should stay as backends or
tool-builder APIs rather than first-class public verbs:

- `runtime.inspection.entries()`
- `runtime.inspection.subscribe(...)`
- `attachInspectionSink(...)`
- `createInspectionBufferSink(...)`
- `graphOf(...)`
- `inspectTransition(...)`
- `inspectMicrosteps(...)`
- `inspectActions(...)`
- `whyNoTransition(...)`
- `formatHarnessTracePretty(...)`
- `formatPendingWorkPretty(...)`
- `formatScenarioTranscript(...)`
- `formatTransactionEventsPretty(...)`

How they stay useful:

- as backends for `behavior`, `story`, and `trace`
- as app-owned tooling surfaces
- as docs/reference surfaces
- as proof/test infrastructure

## Postponed Secondary CLI Surfaces

These are real capabilities, but they should be treated as secondary until the
main codebase-linked workflow is sharp:

- `trace export` -> `exportTraceArtifact(...)`
- `trace import` -> `importTraceArtifact(...)`
- `trace compress` -> `compressTraceArtifact(...)`
- `trace decompress` -> `decompressTraceArtifact(...)`
- any public `inventory` family over raw `.inventory()`

## Thin Composition Code Still Needed

1. Gateway loading for `story ...`
   We need a standard codebase-linked way to load app + story descriptors,
   similar to how `behavior-cli.mjs` loads `BehaviorGateway`.

2. Story discovery and lookup
   The CLI needs one stable story registry path so `story list`,
   `story describe`, `story run`, and `story run --check` can all resolve the same
   story ids.

3. Machine lookup for path discovery
   `story paths ...` needs the same kind of stable codebase-linked machine
   lookup that `story ...` needs for story ids.

4. Trace input normalization
   `trace ...` commands need one adapter that can accept:
   - local inspection proof JSON
   - trace artifact JSON
   - trace output emitted by `story run`

5. Local proof import and slicing
   We have `createLocalInspectionProof(...)`, but not yet one shared reusable
   helper for reading a saved local proof bundle and slicing it by actor,
   correlation, failure set, or timeline.

6. Text renderers for a few structured outputs
   We already have strong behavior and trace formatters, but:
   - `trace diff` still needs a stable text renderer
   - `trace summarize --contextualize` may need a stable human-oriented renderer
   - `trace proof --correlation` may need a thin pretty renderer

7. Structured output contracts
   The main remaining work is deciding exact text and JSON shapes, not building
   new debugging internals.

## New Lower-Level Code That Is Probably Not Needed

Do not add new low-level code for these unless a proven gap appears:

- second story runner
- second expectation-check evaluator
- second story descriptor
- second path engine
- second trace summarizer
- second trace diff engine
- second machine-aware trace analysis engine
- second local proof shape
- second inspection timeline formatter
- second behavior contract builder

## One-Sentence Build Rule

If a new command can be assembled by composing an existing inspect/testing/
runtime primitive plus a stable output contract, do that instead of adding a
new debugging implementation.
